## 4. 架构设计（分层清晰，面试可讲透）

### 整体架构（4层架构，贴合企业级设计）

1. 接入层：任务提交接口（HTTP/gRPC）、任务管理控制台，支持任务创建、编辑、启停、查询。

2. 调度层（核心）：调度中心，负责任务解析、分片分配、依赖编排（DAG）、执行器负载均衡、失败重试。

3. 执行层：执行器集群（K8s 部署），接收调度中心指令，执行具体任务，反馈执行结果。

4. 存储层：Redis（任务队列、分布式锁、延迟任务）、MySQL（任务元数据、执行日志、失败记录）。

架构亮点：去中心化调度（避免单点故障）、执行器集群水平扩展、任务分片并行处理，面试时可重点讲“调度中心如何实现负载均衡”“执行器故障如何容错”。

### 核心功能（面试加分项，重点实现）

- 任务分片：支持任务分片（如百万级数据同步任务，拆分多片并行执行），讲清分片策略（一致性哈希），解决大数据量任务执行效率问题。

- 死信任务处理：任务多次重试失败后，转入死信队列，支持手动重试、批量重试，讲清死信触发条件（超时、执行失败、取消）。

## 4. 核心功能详解（Core Features）

### 4.1 任务类型

- 定时任务：基于 Cron 表达式，支持秒级精度，适用于每日报表、数据备份等场景。

- 延时任务：支持毫秒级延时，适用于订单超时取消、短信延时发送等场景。

- 一次性任务：提交后立即执行，适用于临时数据处理、手动触发的任务。

- DAG 依赖任务：支持任务间依赖编排，如“任务A → 任务B → 任务C”，需等待前序任务执行完成。

### 4.2 任务分片

支持基于一致性哈希的任务分片，将大数据量任务拆分为多个子任务，由多个执行器并行执行，提升任务执行效率。例如：百万级用户积分结算任务，拆分为10个分片，由10个执行器同时执行，缩短执行时间。

### 4.3 幂等性与重试机制

- 幂等性：基于任务唯一ID（UUID）+ 分布式锁，确保任务重复执行不会产生副作用。

- 重试机制：支持自定义重试次数、指数退避策略（如1s、3s、5s、10s重试），失败任务可手动重试或自动转入死信队列。

### 4.4 死信任务处理

任务多次重试失败后，自动转入死信队列，支持手动重试、批量重试、删除操作，同时记录死信原因（超时、执行报错、依赖失败），便于问题排查。

### 4.5 执行器集群

执行器支持集群部署，通过心跳机制与调度中心通信，调度中心根据执行器负载情况分配任务，实现负载均衡；执行器故障时，调度中心自动将任务分配给其他健康节点，保证任务可靠执行。

### 4.6 业务联动（面试重点讲）

无缝对接现有 Go 微服务电商项目，实现以下业务场景：

- 订单超时取消：延时任务，订单创建后30分钟未支付，自动取消订单、释放库存。

- 每日库存核对：定时任务，每天凌晨2点，核对订单库存与实际库存，同步数据差异。

- 销售报表生成：定时任务，每天凌晨3点，统计前一天销售数据，生成报表并推送。

- 用户积分结算：每月月底，批量结算用户积分，生成积分明细。

## 6. 常见问题（FAQ，面试必讲）

- Q1：如何避免任务重复执行？ → A：基于 Redis Redlock 实现分布式锁，结合任务唯一ID，保证幂等性，同时调度中心做任务去重校验。

- Q2：任务分片的策略是什么？ → A：采用一致性哈希分片，根据任务ID哈希分配到不同执行器，保证分片均匀，支持动态扩容。

- Q3：调度中心如何实现高可用？ → A：调度中心集群部署，基于 K8s 实现故障转移，Redis 采用主从架构，MySQL 主从复制，避免单点故障。

- Q4：死信任务的处理逻辑是什么？ → A：任务重试次数达到阈值、执行超时、依赖任务失败，自动转入死信队列，支持手动重试和原因排查。

- Q5：如何对接现有微服务项目？ → A：提供 gRPC/HTTP 接口，可直接在现有项目中调用任务提交接口，实现业务联动。
